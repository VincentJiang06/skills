# Anatomy of an Excellent Industrial Skill

Grounded in the corpus at `learning_from_the_best/` (read 2026-06-04). All paths relative to `/Users/vince/playground/skill-developer/`. The lens: which excellent public skills are reproducible by a deterministic production pipeline (class A — the vince-skill benchmark) versus which are excellent because of irreproducible taste (class B — out of scope).

## 1. The two classes

**Class (A) — heavyweight / industrial.** Multi-file skills whose quality lives in *structure*: routing, bundled scripts, `references/`, evals with verifiable answers, and validators. These are mass-producible because the value is in the apparatus, not the prose.

- `learning_from_the_best/anthropics-skills/skills/mcp-builder/` — 236-line `SKILL.md` orchestrates a 4-phase workflow; offloads 2,537 lines into `reference/` (`evaluation.md` 601, `node_mcp_server.md` 969, `python_mcp_server.md` 718, `mcp_best_practices.md` 249) plus a runnable eval harness `scripts/evaluation.py` + `example_evaluation.xml`.
- `learning_from_the_best/anthropics-skills/skills/pptx/` — script-heavy: `scripts/office/validate.py` validates OOXML against bundled `schemas/ISO-IEC29500-4_2016/*.xsd` with `--auto-repair`; `unpack.py`/`pack.py`/`thumbnail.py`/`soffice.py` are the deterministic engine. `SKILL.md` is mostly a QA harness.
- `learning_from_the_best/anthropics-skills/skills/skill-creator/` — the reference industrial skill: `scripts/` (aggregate_benchmark, run_loop, package_skill, quick_validate), `agents/` (grader/comparator/analyzer subagents), `references/schemas.md`, `eval-viewer/`. An entire eval+benchmark+description-optimization pipeline.
- `learning_from_the_best/anthropics-skills/skills/web-artifacts-builder/` — thin 73-line `SKILL.md` that is essentially a wrapper over two deterministic scripts (`init-artifact.sh`, `bundle-artifact.sh`) + a bundled `shadcn-components.tar.gz`.

**Class (B) — short / flexible / elegant.** A single tight, judgment-driven `SKILL.md`; minimal/no apparatus; output is subjective.

- `learning_from_the_best/anthropics-skills/skills/frontend-design/SKILL.md` — 42 lines, *zero* subfiles. Pure aesthetic direction ("commit to a BOLD aesthetic", "NEVER converge on Space Grotesk"). Quality = taste.
- `learning_from_the_best/superpowers/skills/brainstorming/SKILL.md` — collaborative dialogue ("ask questions one at a time"); a `<HARD-GATE>` and flowchart, but no verifiable output. Scripts only power an optional visual companion.
- `learning_from_the_best/anthropics-skills/skills/canvas-design/SKILL.md` — 129 lines of art-manifesto prose ("museum quality", "meticulously crafted"). Bundles `canvas-fonts/` *assets* but no validator — a hybrid leaning B.

Borderline/instructive: `learning_from_the_best/obsidian-skills/skills/json-canvas/SKILL.md` and `.../obsidian-markdown/SKILL.md` are short but have `references/` and a *verifiable* output contract (valid JSON Canvas / Obsidian syntax), pulling them toward A despite small size.

## 2. Anatomy of an excellent industrial skill

**Description / trigger-line craft (incl. "Use when / Do NOT use").** The `description` is the *only* thing controlling activation — the body is read only after the skill is already live (`workflow-skill-design/.../SKILL.md` principle `description-is-the-trigger`). Best practice: third-person, "Use when…", keyword-dense, and — counterintuitively — it must NOT summarize the workflow, because Claude will then *follow the description instead of reading the body* (`superpowers/skills/writing-skills/SKILL.md` CSO section: a "code review between tasks" summary caused ONE review instead of two). `pptx/SKILL.md`'s description is a masterclass in pushy trigger coverage ("any time a .pptx is involved… even if the extracted content will be used elsewhere"). `skill-creator/SKILL.md` explicitly tells authors to make descriptions "a little bit pushy" to fight under-triggering. "When NOT to use" must name the *alternative* ("use Semgrep for simple pattern matching"), per `workflow-skill-design`. Industrial relevance: triggering is the one thing a pipeline can test deterministically — `skill-creator` ships a whole `run_loop.py` that splits 20 trigger evals 60/40 train/test, runs each 3×, and selects `best_description` by held-out score.

**Progressive disclosure (SKILL.md vs references/ vs scripts).** A three-tier loading model: metadata always in context; `SKILL.md` body on trigger (<500 lines); bundled resources on demand (`skill-creator/SKILL.md` "Anatomy of a Skill"). Measured in corpus: every heavyweight `SKILL.md` obeys the rule (skill-creator 485, mcp-builder 236, pptx 232) while shipping far more behind it (mcp-builder `reference/` = 2,537 lines). Rule enforced as structure, not preference: "one level deep — no reference chains" (`workflow-skill-design` AP-3; trailofbits `CLAUDE.md` "SKILL.md → file1 → file2 is the problem, nested folders are fine"). Domain-variant organization (one reference file per framework, e.g. `node_mcp_server.md` vs `python_mcp_server.md`) lets the agent load only the relevant branch. This is *the* core industrializable lever: it makes context cost bounded and predictable.

**Bundled deterministic scripts.** Move anything mechanical out of the prompt into code. `pptx/scripts/add_slide.py` and `office/validate.py`, `web-artifacts-builder/scripts/init-artifact.sh` (detects Node version, pins Vite, OS-aware `sed`), `mcp-builder/scripts/evaluation.py`. Each script declares a contract in its docstring (add_slide.py documents inputs/outputs precisely). `skill-creator` codifies the *signal* for when to bundle: "if all 3 test cases resulted in the subagent writing a `create_docx.py`, that's a strong signal the skill should bundle that script." Industrial relevance: scripts give reproducible, zero-variance behavior and save tokens on every future invocation — the opposite of re-derived prose.

**Evals / test fixtures.** The defining industrial trait. `mcp-builder/reference/evaluation.md` specifies 10 questions that are Independent, Read-only, Complex, Realistic, **Verifiable by direct string comparison**, and **Stable** (no "current state" counts), with worked good/poor examples and a self-solve verification pass. `skill-creator` runs with-skill vs without-skill (baseline) subagents, grades assertions, and aggregates `benchmark.json` with mean±stddev and delta. `superpowers/skills/writing-skills/SKILL.md` reframes the whole act as TDD: "NO SKILL WITHOUT A FAILING TEST FIRST" — run the baseline scenario, watch it fail, then write the skill. Relevance: an output contract you can check is what lets a pipeline iterate without a human in the loop.

**JSON schemas / contracts.** `skill-creator/references/schemas.md` pins exact shapes for `evals.json`, `grading.json` (fields must be `text`/`passed`/`evidence` — "the viewer reads these field names exactly"), `benchmark.json` (`configuration` must be `"with_skill"`/`"without_skill"`), plus history/comparison/analysis. `json-canvas/SKILL.md` encodes the JSON Canvas 1.0 contract as required-field tables. Relevance: machine-readable contracts are what let separate pipeline stages (engineer → grader → viewer) interoperate without re-negotiation.

**Validators / verification.** Two flavors. (1) *Structural* — `skill-creator/scripts/quick_validate.py` checks frontmatter keys against an allowlist, kebab-case name ≤64 chars, description ≤1024 chars / no angle brackets. (2) *Output* — `pptx/scripts/office/validate.py` validates against XSD and auto-repairs; `json-canvas`/`obsidian-markdown` end with explicit "Validation Checklist" steps (all IDs unique, every edge references a real node). (3) *Visual/process* — `pptx/SKILL.md` mandates a subagent QA loop ("Assume there are problems… if you found zero issues you weren't looking hard enough", re-verify after each fix). Relevance: verification is the gate that converts "looks done" into "is done" — `verification-before-completion` as a hard requirement.

**Packaging.** `skill-creator/scripts/package_skill.py` **validates first, then zips** to a `.skill` file, excluding `__pycache__`/`node_modules`/`evals/`. Plugin packaging is standardized: `.claude-plugin/plugin.json`, components at plugin root not inside `.claude-plugin/`, registration in root `marketplace.json` (trailofbits `CLAUDE.md`). Vercel uses `.zip` distribution (README map). Relevance: a repeatable build/exclude/validate step makes mass output shippable.

**Meta patterns (self-checking, gates).** Hook-driven loop gates: `trailofbits-skills/plugins/skill-improver/hooks/stop-hook.sh` blocks Stop and re-prompts until the assistant emits a whole-line `<skill-improvement-complete>` marker (natural language won't stop it), bounded by `max_iterations`. Severity-tiered self-review (`skill-improver/SKILL.md`: Critical/Major/Minor with "evaluate minor before fixing"). Numbered phases with **entry/exit criteria** (`workflow-skill-design` principle `numbered-phases`; `spec-driven-development/SKILL.md`'s gated SPECIFY→PLAN→TASKS→IMPLEMENT, each human-reviewed). "Rationalizations to Reject" / "Red Flags" tables to harden discipline skills against loopholes (`writing-skills`, `spec-driven-development`). The **10,000-file test** for tool-call scalability (`workflow-skill-design` principle `scalable-tool-patterns`, AP-18/19). Relevance: these are exactly the reproducible control structures a conductor pipeline can template.

## 3. The official format baseline

The spec (`anthropics-skills/spec/agent-skills-spec.md`) now just points to `agentskills.io/specification`; the `template/SKILL.md` is the true minimum: YAML frontmatter with **`name` + `description` only**, then markdown. Allowed frontmatter keys (per `quick_validate.py`): `name, description, license, allowed-tools, metadata, compatibility`; name kebab-case ≤64, description ≤1024 and no `<>`. Everything else is convention the best skills *add on top*: progressive disclosure (`scripts/`/`references/`/`assets/`), mandatory "When to Use / When NOT to Use" (trailofbits + addyosmani `CLAUDE.md` require both sections), `allowed-tools` least-privilege, evals + schemas + validators + packaging. So the spec is permissive; industrial-grade = spec + the apparatus in §2.

## 4. Why class (B) resists industrialization

- **Quality is taste, not a checkable contract.** `frontend-design/SKILL.md` asks for an "UNFORGETTABLE" aesthetic and forbids convergence ("NEVER … Space Grotesk"); `canvas-design/SKILL.md` wants "museum quality… meticulously crafted." There is no string-comparison answer and no XSD — exactly what `skill-creator` flags ("subjective skills (writing style, design quality) are better evaluated qualitatively — don't force assertions"). A deterministic loop has nothing to optimize against; correctness collapses to human judgment per output. `mcp-builder`'s eval recipe (verifiable, stable, single-answer) is structurally impossible here.
- **The work is open-ended dialogue / immersion, not a pipeline.** `brainstorming/SKILL.md` is "ask questions one at a time," adapt to the human, decide *per question* whether a browser helps — an interactive, branching collaboration. You cannot pre-script the branch taken; the skill's value is responsiveness, which a fixed process flattens. (Even superpowers say so: their `CLAUDE.md` rejects "compliance" restructurings of behavior-shaping skills "without extensive eval evidence" — the tuning is empirical and hand-held, not generated.)
- **Excellence comes from compression and restraint a generator over-builds.** frontend-design is 42 lines doing more than many 400-line skills; its power is what it *omits*. `writing-skills` warns "keep the prompt lean… one excellent example beats many." A mass-production pipeline's instinct is to add scaffolding (more rules, more references, more gates) — which is additive value for class A but actively *dilutes* class B. Vince's own memory (`feedback-prefers-simple-scoped-skills`) names this tension: the lean direction often wins, and it is the hardest thing to manufacture.

## 5. Scoring rubric — is this skill industrial-grade?

A yes/no checklist derived from §2–3. ~8+ "yes" (and all of 1–3) ≈ industrial-grade; heavily-"no" but excellent ⇒ probably class B (judge by taste, don't try to mass-produce).

1. [ ] `SKILL.md` has valid frontmatter (`name` kebab-case ≤64; `description` ≤1024, no `<>`) and passes a structural validator.
2. [ ] Description is third-person, "Use when…", keyword-rich, and does **not** summarize the workflow.
3. [ ] Has explicit "When to Use" **and** "When NOT to Use" (the latter naming concrete alternatives).
4. [ ] `SKILL.md` body < 500 lines, with detail pushed to `references/` (progressive disclosure).
5. [ ] No reference chains — every linked file is one hop from `SKILL.md`; no hardcoded absolute paths.
6. [ ] Bundles deterministic `scripts/` for the mechanical/repeated work, each with a documented input/output contract.
7. [ ] Ships evals/test fixtures with **verifiable, stable** expected outputs (string-comparable or schema-checkable).
8. [ ] Defines machine-readable JSON schemas/contracts for any data exchanged between stages.
9. [ ] Has a validator or explicit verification step that gates "done" (structural, output-schema, or QA loop).
10. [ ] Workflow uses numbered phases with entry/exit criteria (or a named pattern: routing/pipeline/linear/safety-gate/task-driven).
11. [ ] `allowed-tools` follows least privilege (read-only skills have no Write/Bash).
12. [ ] Tool-call patterns scale (passes the 10,000-file test; no Cartesian-product greps or unbounded subagent spawning).
13. [ ] Discipline/audit skills include a "Rationalizations to Reject" / "Red Flags" hardening table.
14. [ ] Has a repeatable packaging step (validate-then-bundle to `.skill`/plugin, with build artifacts excluded).
