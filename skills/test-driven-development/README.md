# test-driven-development

> 对*非平凡*行为做测试驱动开发 —— 测试套件是当前目标的「活规格」，目标变了就改它，而不只是一味新增。

[English](README.en.md) · **简体中文**

**做什么** —— 对*非平凡*行为做测试驱动开发：先写或改一个会失败的测试、看它失败一次，再写最小实现使其通过。测试套件被当作当前目标的**活规格** —— 目标变了就改 / 合并 / 删测试，而不只是一味新增。

**好在哪** ——
- 一个有判别力的**适度门**治好过度触发：只在真逻辑 / 改 bug / 改已测行为时介入；重命名、配置常量、抛弃式试验、生成代码、纯文档一律跳过。
- 一个**修改模式**优先改 / 合并 / 删除而非堆叠 —— 一个功能组一个测试，绝不增殖。
- 把清点套件、跑测试、扫陈旧测试派给 subagent，主线保持轻快。

**什么时候用** —— 边写边自动触发：实现真逻辑（带分支 / 边界的函数·方法·端点·组件）·修 bug（先用失败测试复现）·改已测代码的行为；也可用 `/test-driven-development`（简称 TDD）显式调用。
**不适用** —— 平凡 / 机械改动（重命名、格式化、注释、纯类型、配置常量微调）；抛弃式原型 / 试验；生成代码；纯文档改动。
