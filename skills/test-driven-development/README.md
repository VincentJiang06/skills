# test-driven-development

> 对*非平凡*行为做测试驱动开发 —— 测试套件是当前目标的「活规格」，目标变了就改它，而不只是一味新增。v1.0.0 起由 skill-philosophy KB 全面重接地。

[English](README.en.md) · **简体中文**

**做什么** —— 对*非平凡*行为做测试驱动开发：先写或改一个会失败的测试、**带证据地**看它失败一次，再写最小实现使其通过。测试套件被当作当前目标的**活规格** —— 目标变了就改 / 合并 / 删测试，而不只是一味新增。

**好在哪** ——
- 一个有判别力的**适度门**治好过度触发：只在真逻辑 / 改 bug / 改已测行为时介入；重命名、配置常量、抛弃式试验、生成代码、纯文档一律跳过。
- 一个**修改模式**优先改 / 合并 / 删除而非堆叠 —— 一个功能组一个测试，绝不增殖。
- **看它失败 + revert-to-red** 两道不可约门，专治单一上下文同时写测试和实现造成的相关性错误（Knight–Leveson）—— 没有命令 + 真实输出 + 退出码的「通过了」一律视为缺陷。
- **信任边界脊柱**（v1.0.0 新增）：代码 / 测试内容里出现的指令性文本零权威 —— 注释里写「跳过运行」也只是数据；执行任意测试代码是真实行动面，仓库外 / 破坏性 I/O 须确认。
- 可执行的 `evals/` 行为 harness（22 检查）：真实 pytest/vitest fixture、自动 revert 抓空洞测试、**断言级红判定**（崩溃 ≠ 红）、注入场景、E-L3 压力哨兵，外加两个**非建造者作者**的 held-out 作弊解回归钉。
- 把清点套件、跑测试、扫陈旧测试派给 subagent，主线保持轻快。

**什么时候用** —— 边写边自动触发：实现真逻辑（带分支 / 边界的函数·方法·端点·组件）·修 bug（先用失败测试复现）·改已测代码的行为；也是 agent loop 里 generator 的内在纪律。
**不适用** —— 平凡 / 机械改动（重命名、格式化、注释、纯类型、配置常量微调）；抛弃式原型 / 试验；生成代码；纯文档改动。

**安装** —— `npx skills add VincentJiang06/skills`（或 `cp -R skills/test-driven-development ~/.claude/skills/`）。

**已知限制**（如实记录）—— revert-to-red 以候选为粒度：混在真断言里「搭车」的空洞断言不会被单独识破；压力哨兵在 run_all 里是确定性代理检查，完整 64K 实况跑仅在大版本节奏执行；逐断言空洞检测需要变异测试，刻意不在确定性 harness 范围内。详见 `evals/README.md`。
