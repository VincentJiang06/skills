# Loop runbook — Migrate ~40 independent Node modules from CommonJS to ESM in parallel, then land a single cutover (package type:module + build/test config), without regressing any module's tests

> A runnable closed-loop design generated by loop-constructor. Follow it as a loop: make a small change → run the stage check → read the result → fix → repeat until the check passes, then advance.
> **Altitude:** large — ~40 largely-independent modules can be converted concurrently with no shared-write conflict (each module + its tests is a disjoint surface), and the scale makes fan-out a real speedup over sequential. Parallelism = concurrent codex exec processes in separate git worktrees coordinating via an on-disk ledger — large, not medium.

## Hard constraints
- Minimal viable change; don't touch unrelated files.
- Don't change public APIs / add deps without justification.
- No production resources.
- On a requirement contradiction → STOP and report.

## How to run this loop (Codex CLI)
> This runbook targets the OpenAI Codex CLI (single-agent, `codex exec`). Each stage is one or more `codex exec` invocations; every invocation is a fresh process/context — durable state lives on disk (see Harness primitives), never in a chat history.
- **Drive each stage** with `codex exec "<stage prompt>"` in the project root (sandbox: `workspace-write`; approval policy per your risk posture). Standing project instructions go in `AGENTS.md`, not in the prompt.
- **The gate** is that stage's **Check** command below — run it after each pass; advance only on green. On a fail, retry within the stage's cap, then take its on-failure route.
- **Evaluator = a SEPARATE `codex exec`** handed ONLY the diff + the Contract below (never this transcript or the generator's reasoning), told the artifact is broken and to prove it — e.g. `codex exec "$(cat .loop/prompts/evaluator.md)" --sandbox read-only`. Do not let the generating process grade itself.
- **State lives on disk.** On a `codex resume` (or any rebuilt context) re-read the on-disk state first; never trust an in-context summary. Capture a run log with `codex exec … > .loop/run-log.txt 2>&1`.

## Roles (three separate contexts — never one agent grading itself)
- **Planner:** enumerate the module graph, shard the ~40 modules into disjoint independent sets, write the manifest + contract; never convert a module
- **Generator:** in each shard worktree, convert that shard's modules to ESM and keep their tests green; may NOT touch another shard's files, weaken a test, or grade its own shard
- **Evaluator (fresh context, adversarial — told the artifact is broken, proves it):** a fresh read-only codex exec that never saw any shard's conversion; given only the merged diff + contract.md; assume the migration is broken and prove it — run the whole suite, run the repo-wide residual scan, hunt extensionless imports and a shared helper converted two ways, and check the manifest covered every module

## Contract (negotiated acceptance criteria — grade THIS, not the original spec)
> The generator and evaluator agreed these testable assertions before the build. The loop is done only when every one is proven by its check.
- **`A1`** (stage `characterize`) — the whole suite is green on the untouched CJS baseline and pinned as tag cjs-baseline · graded by `npm test`
- **`A2`** (stage `characterize`) — the manifest lists EVERY module containing a CJS construct (no module silently omitted) · graded by `node scripts/cjs-residual-scan.mjs --audit-manifest .loop/modules.json`
- **`A3`** (stage `convert_shard_a`) — every shard-A module is ESM with green ESM tests and zero residual CJS · graded by `npm run test:shard -- a && node scripts/cjs-residual-scan.mjs --shard a --expect-zero`
- **`A4`** (stage `convert_shard_b`) — every shard-B module is ESM with green ESM tests and zero residual CJS · graded by `npm run test:shard -- b && node scripts/cjs-residual-scan.mjs --shard b --expect-zero`
- **`A5`** (stage `convert_shard_c`) — every shard-C module is ESM with green ESM tests and zero residual CJS · graded by `npm run test:shard -- c && node scripts/cjs-residual-scan.mjs --shard c --expect-zero`
- **`A6`** (stage `cross-cutting`) — no shard's worktree edited another shard's files (disjoint surfaces) · graded by `node scripts/assert-disjoint-worktrees.mjs .loop/modules.json`
- **`A7`** (stage `cutover`) — after merge, package.json has type:module and the build passes under ESM · graded by `npm run build`
- **`A8`** (stage `cutover`) — the whole suite passes under type:module (cross-shard breakage surfaces) · graded by `npm test`
- **`A9`** (stage `cutover`) — a repo-wide residual scan finds zero CJS constructs anywhere, including files no shard owned · graded by `node scripts/cjs-residual-scan.mjs --all --expect-zero`
- **`A10`** (stage `cutover`) — no relative import is extensionless (ESM requires explicit .js) · graded by `node scripts/cjs-residual-scan.mjs --assert-explicit-extensions`

## Why this shape (decision log)
> The D0–D6 selection procedure that derived this loop's shape.
- **D0** (loop) — per-module done is machine-decidable: that module's unit tests pass under ESM AND a residual scan (AST/grep for require/module.exports/__dirname) finds zero CJS constructs in it; final cutover is decidable by the whole suite + build exit code. A runnable check answers done ⇒ loop.
- **D1** (staged (2 seam types)) — seam A: characterize hands a GREEN whole-suite baseline (tag `cjs-baseline`) + a per-module manifest to the conversion shards. seam B: every conversion shard hands a converted+green module set to cutover, which flips package.json/config and runs the whole suite. Each conversion shard's check runs without cutover existing; cutover consumes the converted modules unchanged; shards edit disjoint files ⇒ boundaries pass the seam test.
- **D2** (characterize=explore_narrow, convert_shard_*=retry, cutover=plan_execute_verify) — characterize enumerates the module graph + pins the baseline (explore_narrow); each shard is a mechanical, retryable per-module transform gated by tests + a residual scan (retry); cutover is a one-shot risky flip needing mid-flight correction across the whole build (plan_execute_verify).
- **D3** (on_the_loop) — the migration is reversible per worktree (discard a bad worktree) and each shard's check discriminates (tests + residual scan), so a wrong shard is caught at its gate ⇒ human reviews at the merge/cutover gates, not each per-module iteration. cutover is the one high-blast step and is reviewed before it lands.
- **D4** (large) — the ~40 modules are largely independent — there exist many stage-pairs (the conversion shards) with NO dependency path between them that benefit from running as separate concurrent agents with no shared-write conflict. Realized as concurrent codex exec processes, each in its own git worktree, coordinating only through a shared on-disk ledger; a final gate stage merges the worktrees and runs the whole-suite check.
- **D5** (per-shard caps + restart routing + escalate-on-cutover-conflict) — each conversion shard cap 8, on_failure restart (discard the shard's worktree and re-derive that module set from the manifest — a half-converted module is archaeology, a clean re-transform is faster); cutover cap 3 loopback-&gt;characterize if the merge reveals a mis-pinned baseline; outer escalate on an unresolvable cross-module import cycle that blocks independent conversion.
- **D6** (iteration_first) — per-module tests + residual scan are fast/cheap, so each shard runs many short passes (one module per pass, cap 8) with near-free feedback. cutover is the one completeness-first exception (cap 3, thorough whole-suite+build check) since re-entering a whole-repo flip is expensive — recorded as a per-stage cadence exception.

## Stages (run in dependency order; each stage is its own gated loop)
> After a stage gate goes green, commit/checkpoint it; on a loopback, reset to the last green checkpoint rather than editing forward.

### Stage 1 — `characterize`  ·  pattern: explore_narrow

- **Done when:** the whole test suite passes on the untouched CJS baseline, is pinned as git tag `cjs-baseline`, and a per-module manifest (.loop/modules.json listing each module + its test command) is written
- **Check (run — the gate):** `npm test && test -s .loop/modules.json && git tag -f cjs-baseline`  ·  expect: `pass`
- **Falsifiable when:** the baseline suite is red, or .loop/modules.json is empty/missing — the check exits non-zero
- **Passing-but-wrong (ruled out):** the manifest lists only a subset of modules so untracked modules are silently never converted -&gt; ruled out: the manifest is generated by scanning for every file containing a require()/module.exports and the count is asserted against `git ls-files '*.js' | xargs grep -l` so a short manifest fails
- **Depends on:** — (entry stage)  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - baseline suite cannot be made green on the untouched tree (environment broken)
- **On failure:** escalate.

### Stage 2 — `convert_shard_a`  ·  pattern: retry

- **Done when:** every module assigned to shard A is ESM (import/export) with its unit tests green under ESM and a residual scan finding zero CJS constructs in shard A's files
- **Check (run — the gate):** `npm run test:shard -- a && node scripts/cjs-residual-scan.mjs --shard a --expect-zero`  ·  expect: `pass`
- **Falsifiable when:** a shard-A module still contains require()/module.exports/__dirname/__filename, or a shard-A test fails under ESM -&gt; non-zero exit
- **Passing-but-wrong (ruled out):** the residual scan greps only `require(` and misses `module.exports`, dynamic require, __dirname, or `import x = require()` -&gt; ruled out: the scan is AST-based and enumerates the whole CJS class (require call, module.exports/exports assignment, __dirname/__filename, createRequire) not a single token
- **Depends on:** `characterize`  ·  admit downstream only once this check passes.
- **Stop:** max 8 iterations; failure branches:
    - a shard-A module cannot be converted without touching another shard's files (hidden coupling)
- **On failure:** restart.

### Stage 3 — `convert_shard_b`  ·  pattern: retry

- **Done when:** every module assigned to shard B is ESM with its unit tests green under ESM and a residual scan finding zero CJS constructs in shard B's files
- **Check (run — the gate):** `npm run test:shard -- b && node scripts/cjs-residual-scan.mjs --shard b --expect-zero`  ·  expect: `pass`
- **Falsifiable when:** a shard-B module still contains a CJS construct, or a shard-B test fails under ESM -&gt; non-zero exit
- **Passing-but-wrong (ruled out):** a module is 'converted' but its test was never run under the ESM loader (still executed via the CJS jest transform) -&gt; ruled out: test:shard runs node with --experimental-vm-modules / the ESM test runner so a module that only works under CJS fails
- **Depends on:** `characterize`  ·  admit downstream only once this check passes.
- **Stop:** max 8 iterations; failure branches:
    - a shard-B module cannot be converted without touching another shard's files (hidden coupling)
- **On failure:** restart.

### Stage 4 — `convert_shard_c`  ·  pattern: retry

- **Done when:** every module assigned to shard C is ESM with its unit tests green under ESM and a residual scan finding zero CJS constructs in shard C's files
- **Check (run — the gate):** `npm run test:shard -- c && node scripts/cjs-residual-scan.mjs --shard c --expect-zero`  ·  expect: `pass`
- **Falsifiable when:** a shard-C module still contains a CJS construct, or a shard-C test fails under ESM -&gt; non-zero exit
- **Passing-but-wrong (ruled out):** an import path drops the required .js extension (ESM needs explicit extensions) but the stale CJS resolver still finds it during the shard test -&gt; ruled out: the shard test runs under the pure ESM loader where extensionless relative imports throw ERR_MODULE_NOT_FOUND
- **Depends on:** `characterize`  ·  admit downstream only once this check passes.
- **Stop:** max 8 iterations; failure branches:
    - a shard-C module cannot be converted without touching another shard's files (hidden coupling)
- **On failure:** restart.

### Stage 5 — `cutover`  ·  pattern: plan_execute_verify

- **Done when:** all shard worktrees are merged, package.json has "type":"module", build/test config is ESM, and the WHOLE suite + build pass with a repo-wide residual scan finding zero CJS constructs
- **Check (run — the gate):** `npm run build && npm test && node scripts/cjs-residual-scan.mjs --all --expect-zero`  ·  expect: `pass`
- **Falsifiable when:** the merged tree fails the build, any test fails under type:module, or the repo-wide residual scan finds a leftover CJS construct anywhere -&gt; non-zero exit
- **Passing-but-wrong (ruled out):** two shards each converted a shared helper differently and the merge silently kept one, breaking the other's callers -&gt; ruled out: cutover runs the WHOLE suite (not per-shard) so a cross-shard breakage surfaces, and the repo-wide residual scan covers files no shard owned
- **Depends on:** `convert_shard_a`, `convert_shard_b`, `convert_shard_c`  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - merged tree cannot build under type:module without reverting a shard
    - a cross-shard conflict cannot be resolved without re-sharding
- **On failure:** loop back to `characterize`.

## Orchestration (large altitude)
> Stages with no dependency path between them run as **separate concurrent `codex exec` processes**, each in its own git worktree, coordinating only through the shared on-disk ledger — Codex is single-agent, so fan-out is N concurrent OS processes, not in-process spawning. See loop-principle `pattern.multi_agent_orchestra` + `templates/multi_agent_plan.template.json` for roles / isolation / the shared-state ledger.

## Stop conditions (overall)
- **Success:** characterize is green and pinned, all three conversion shards pass their per-shard check in their own worktrees, and cutover merges them with the whole suite + build + repo-wide residual scan all green
- **Max iterations (budget cap):** 4
- **Failure branches:**
  - any shard exhausts its max_iterations without converting cleanly
  - cutover loops back to characterize twice without converging
- **Escalate to a human when:**
  - an unresolvable cross-module import cycle blocks independent sharding
  - the sharding itself is found wrong (a 'module' spans two shards' surfaces)

## Terminal states
- **SUCCESS** — characterize is green and pinned, all three conversion shards pass their per-shard check in their own worktrees, and cutover merges them with the whole suite + build + repo-wide residual scan all green → done.
- **STOPPED_UNMET** — a max-iteration cap is hit or a failure branch fires: stop, report the last green stage and the unmet gate, hand back the partial diff, do NOT merge. Never relax a gate to force a pass.
- **ESCALATE** — one of the escalate triggers above fires: hand to a human.

## Human placement
- on_the_loop

## Checker (separate, fresh-context)
- a fresh-context reviewer sees only the merged diff + the whole-suite check + the repo-wide residual scan; flags any cross-shard regression, an extensionless import, or a module missing from the manifest — not per-module style

## Risk guards
- **shared-write conflict between concurrent shards** → each shard runs in its own git worktree editing only its manifest-assigned files; A6 asserts disjoint surfaces; coordination is only via the on-disk ledger
- **reward hacking (a module marked converted but its test never ran under ESM)** → test:shard runs under the ESM loader so a CJS-only module fails; the residual scan is AST-based over the whole CJS class
- **error amplification across the merge (two shards convert a shared helper differently)** → cutover runs the WHOLE suite not per-shard, and a repo-wide residual scan covers unowned files; a cross-shard break surfaces before cutover lands
- **premature over-delegation / archaeology in a stuck shard** → a stuck shard restarts (discard its worktree, re-derive from the manifest) rather than patching a half-converted pile
- **permission blast radius** → shards run --sandbox workspace-write in isolated worktrees; the evaluator runs --sandbox read-only; no host access beyond the repo, never danger-full-access

## Harness primitives
- git worktree per conversion shard (git worktree add ../wt-shard-&lt;x&gt;) so concurrent codex exec processes never share a working tree
- shared on-disk ledger .loop/run-state.json — which shards are done, each shard's last green commit, the merge order — the ONLY cross-process channel (never shared context)
- git tag cjs-baseline as the immutable regression + audit reference
- .loop/modules.json manifest (module -&gt; shard -&gt; test command) the planner writes and shards read
- contract.md holding the negotiated assertions
- AGENTS.md standing instruction: each codex exec re-reads .loop/run-state.json + its shard slice of the manifest before acting
