# Loop runbook — Add token-bucket rate-limiting middleware to an Express JSON API, config-driven from config/ratelimit.json, covered by tests, without regressing the existing endpoint contract

> A runnable closed-loop design generated by loop-constructor. Follow it as a loop: make a small change → run the stage check → read the result → fix → repeat until the check passes, then advance.
> **Altitude:** medium — Multi-file change (middleware + config loader + tests) on a public request path, but reversible, dev-only, and single-agent. Stages are strictly sequential (each consumes the prior stage's stable artifact), so sequential gated stages with no parallel fan-out — medium, not large.

## Hard constraints
- Minimal viable change; don't touch unrelated files.
- Don't change public APIs / add deps without justification.
- No production resources.
- On a requirement contradiction → STOP and report.

## How to run this loop (Codex CLI)
> This runbook targets the OpenAI Codex CLI (single-agent, `codex exec`). Each stage is one or more `codex exec` invocations; every invocation is a fresh process/context — durable state lives on disk (see Harness primitives), never in a chat history.
- **Drive each stage** with `codex exec "<stage prompt>"` in the project root (sandbox: `workspace-write`; approval policy per your risk posture). Standing project instructions go in `AGENTS.md`, not in the prompt.
- **The gate** is that stage's **Check** command below — run it after each pass; advance only on green. On a fail, retry within the stage's cap, then take its on-failure route.
- **Evaluator = a SEPARATE `codex exec`** handed ONLY the diff + the Contract below (never this transcript or the generator's reasoning), told the artifact is broken and to prove it — e.g. `codex exec "$(cat .loop/prompts/evaluator.md)" --sandbox read-only`. Do not let the generating process grade itself.
- **State lives on disk.** On a `codex resume` (or any rebuilt context) re-read the on-disk state first; never trust an in-context summary. Capture a run log with `codex exec … > .loop/run-log.txt 2>&1`.

## Roles (three separate contexts — never one agent grading itself)
- **Planner:** turn the goal into the staged spec + the negotiated contract; write config/ratelimit.json's schema expectations; never write middleware or test code
- **Generator:** write the middleware + config loader + tests to satisfy the contract; may NOT weaken a test, grade its own output, or hardcode the threshold
- **Evaluator (fresh context, adversarial — told the artifact is broken, proves it):** a fresh codex exec that never saw the impl or the generator's reasoning; given only the diff + contract.md; assume the limiter is broken and prove it — send threshold+1 requests and check for 429, edit config/ratelimit.json and confirm the cutover moves, race two clients, and diff test/ against the baseline tag for any loosened assertion

## Contract (negotiated acceptance criteria — grade THIS, not the original spec)
> The generator and evaluator agreed these testable assertions before the build. The loop is done only when every one is proven by its check.
- **`A1`** (stage `characterize`) — every pre-existing public endpoint returns its captured baseline status + body (no regression) · graded by `npm test api:contract`
- **`A2`** (stage `implement`) — the (threshold+1)th request inside the window returns HTTP 429; every request at or below the threshold returns 200 · graded by `npm test api:ratelimit -- -t threshold-boundary`
- **`A3`** (stage `implement`) — the 429 response carries a Retry-After header with a positive integer seconds value · graded by `npm test api:ratelimit -- -t retry-after`
- **`A4`** (stage `implement`) — the limiter is keyed per client so one client cannot exhaust another client's quota · graded by `npm test api:ratelimit -- -t per-client-isolation`
- **`A5`** (stage `implement`) — the threshold and window come from config/ratelimit.json — changing the config file changes the observed cutover, no hardcoded limit · graded by `npm test api:ratelimit -- -t config-driven`
- **`A6`** (stage `implement`) — a malformed config/ratelimit.json fails fast at startup with a clear error rather than silently disabling the limiter · graded by `npm test api:ratelimit -- -t config-malformed`
- **`A7`** (stage `implement`) — the window resets: after the window elapses, a client is served 200 again · graded by `npm test api:ratelimit -- -t window-reset`
- **`A8`** (stage `verify`) — the full contract suite and full ratelimit suite pass together at the end · graded by `npm test api:contract && npm test api:ratelimit`
- **`A9`** (stage `verify`) — no test assertion was weakened relative to the pinned baseline over the whole run · graded by `git diff --exit-code ratelimit-baseline -- test/`

## Why this shape (decision log)
> The D0–D6 selection procedure that derived this loop's shape.
- **D0** (loop) — runnable checks answer done without a human reading output: `npm test api:contract` (regression), `npm test api:ratelimit` (429/200/Retry-After/per-client), and a config-load assertion — done is machine-decidable, so it is a loop.
- **D1** (staged (2 seams)) — seam A: characterize hands a GREEN captured contract/regression suite to implement (implement must keep it green); seam B: implement hands a working 429 middleware + loaded config to verify. Each handoff is a stable, checkable artifact the upstream can prove without the downstream existing, consumed unchanged, non-conflicting surfaces -&gt; both boundaries pass the seam test -&gt; staged with 3 stages.
- **D2** (characterize=explore_narrow, implement=plan_execute_verify, verify=retry) — characterize searches/pins the existing contract (unknown extent -&gt; explore_narrow); implement is a risky middleware change on the request path needing mid-flight correction (plan_execute_verify); verify is a mechanical retryable re-assertion of the full contract + ratelimit suite (retry).
- **D3** (on_the_loop) — dev-only, reversible, and the checks discriminate well (ratelimit test asserts 200-below AND 429-above, per-client keying) so a wrong iteration is caught by a check, not silently shipped -&gt; human reviews at the gates, not every iteration.
- **D4** (medium) — the three stages form a strict dependency chain (characterize -&gt; implement -&gt; verify); there is no pair of independent stages that could run concurrently and benefit from fan-out -&gt; sequential single-agent, not large.
- **D5** (per-stage caps + loopback routing + escalate-on-public-change) — characterize cap 3 escalate (a red baseline is an environment problem, not our change); implement cap 8 loopback-&gt;characterize (a leaked regression means the baseline needs re-pinning); verify cap 5 loopback-&gt;implement; outer escalate on any needed public-API/response-schema change.
- **D6** (iteration_first) — the gate checks are fast/cheap (npm test in seconds), so frequent small passes give near-free feedback and beat few long ones. Tuned accordingly: implement cap raised to 8, verify pattern is retry (cheap re-attempt), per-pass scope kept to one slice (add one ratelimit behavior at a time). characterize stays low-cap because it is a one-shot pinning, not iterative.

## Stages (run in dependency order; each stage is its own gated loop)
> After a stage gate goes green, commit/checkpoint it; on a loopback, reset to the last green checkpoint rather than editing forward.

### Stage 1 — `characterize`  ·  pattern: explore_narrow

- **Done when:** the existing public-endpoint contract suite passes on the untouched baseline and is captured (as a git tag `ratelimit-baseline`) as the regression set later stages must keep green
- **Check (run — the gate):** `npm test api:contract && git tag -f ratelimit-baseline`  ·  expect: `pass`
- **Falsifiable when:** any existing endpoint returns a different status or body than the captured baseline expects -&gt; the contract suite exits non-zero
- **Passing-but-wrong (ruled out):** a stale/empty contract suite that vacuously passes with zero assertions -&gt; ruled out: the suite asserts a concrete status+body per endpoint and CI fails the suite if it collects zero tests
- **Depends on:** — (entry stage)  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - contract suite cannot be made green on the untouched baseline (environment broken, not our change)
- **On failure:** escalate.

### Stage 2 — `implement`  ·  pattern: plan_execute_verify

- **Done when:** token-bucket middleware reads threshold+window from config/ratelimit.json and returns 429 past the threshold with a Retry-After header, per client, while the captured contract regression set stays green
- **Check (run — the gate):** `npm test api:ratelimit && npm test api:contract`  ·  expect: `pass`
- **Falsifiable when:** the (threshold+1)th request in the window still returns 200 (no 429), or the config values are ignored (a changed config/ratelimit.json threshold does not change the observed cutover), or any contract test regresses -&gt; a test exits non-zero
- **Passing-but-wrong (ruled out):** middleware returns 429 for EVERY request (even below the threshold), passing a naive 'a 429 appeared' test -&gt; ruled out: the ratelimit suite asserts 200 at/below the threshold AND 429 only above it, AND drives the threshold from config so a hardcoded limit fails the config-parametrized test
- **Depends on:** `characterize`  ·  admit downstream only once this check passes.
- **Stop:** max 8 iterations; failure branches:
    - 3 consecutive same-type failures on the ratelimit boundary
    - the contract regression set goes red (a behavior change leaked into an existing endpoint)
- **On failure:** loop back to `characterize`.

### Stage 3 — `verify`  ·  pattern: retry

- **Done when:** the full contract suite + the full ratelimit suite pass together, and no test assertion was weakened relative to the ratelimit-baseline tag
- **Check (run — the gate):** `npm test api:contract && npm test api:ratelimit && git diff --exit-code ratelimit-baseline -- test/`  ·  expect: `pass`
- **Falsifiable when:** any contract or ratelimit test fails together, OR a test file was weakened since the baseline tag (git diff finds a change under test/) -&gt; non-zero exit
- **Passing-but-wrong (ruled out):** the agent loosens a ratelimit assertion so both suites go green -&gt; ruled out: `git diff --exit-code ratelimit-baseline -- test/` fails on ANY test change since the pinned baseline tag, and the tag was captured in stage 1 before any code landed
- **Depends on:** `implement`  ·  admit downstream only once this check passes.
- **Stop:** max 5 iterations; failure branches:
    - suites cannot both pass without weakening a test
    - a test file diverged from the baseline tag and cannot be justified
- **On failure:** loop back to `implement`.

## Stop conditions (overall)
- **Success:** characterize, implement, and verify each pass their check in dependency order, the diff stays within the middleware + config loader + tests, and no test diverged from the ratelimit-baseline tag
- **Max iterations (budget cap):** 4
- **Failure branches:**
  - any stage exhausts its max_iterations without passing its check
  - verify forces a redesign of implement twice without converging
- **Escalate to a human when:**
  - a public-API or response-schema change becomes necessary to meet the limit
  - the contract itself is found to be wrong (an assertion contradicts a real requirement)

## Terminal states
- **SUCCESS** — characterize, implement, and verify each pass their check in dependency order, the diff stays within the middleware + config loader + tests, and no test diverged from the ratelimit-baseline tag → done.
- **STOPPED_UNMET** — a max-iteration cap is hit or a failure branch fires: stop, report the last green stage and the unmet gate, hand back the partial diff, do NOT merge. Never relax a gate to force a pass.
- **ESCALATE** — one of the escalate triggers above fires: hand to a human.

## Human placement
- on_the_loop

## Checker (separate, fresh-context)
- a fresh-context reviewer sees only the diff + the per-stage checks + config/ratelimit.json; flags any behavior change to an existing endpoint, a hardcoded limit that ignores config, or a weakened test — not general style

## Risk guards
- **reward hacking / test overfitting (agent weakens a ratelimit test to pass)** → the verify stage diffs test/ against the pinned baseline tag; any test change since baseline fails the gate
- **config ignored / hardcoded threshold passing a naive test** → A5 drives the threshold from config/ratelimit.json so a hardcoded limit fails the config-parametrized test
- **error amplification (a wrong characterize baseline poisons later gates)** → stage gates run in dependency order; a leaked regression loops implement back to characterize to re-pin, rather than head-banging in place
- **permission blast radius** → dev-only; no production resources; evaluator runs --sandbox read-only, generator --sandbox workspace-write

## Harness primitives
- git tag `ratelimit-baseline` pinned in stage 1 as the immutable regression reference
- external scratch ledger (.loop/run-state.md) holding which stage is done, the last green checkpoint, and what's next across context resets
- contract.md holding the negotiated assertions (the graded criteria)
- AGENTS.md standing instruction: each codex exec re-reads .loop/run-state.md + contract.md before acting
