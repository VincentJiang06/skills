# Loop runbook — Add completeness-first iteration-profile guidance (D6) to loop-constructor, dogfood it, and harden via a 3-lens attacker review — run as a completeness-first loop (few long thorough passes).

> A runnable closed-loop design generated by loop-constructor. Follow it as a loop: make a small change → run the stage check → read the result → fix → repeat until the check passes, then advance.
> **Altitude:** medium — A doc/contract change to one skill, fully reversible (git), single-agent. The attacker fan-out lives INSIDE the review stage, not as parallel debug stages — so medium, not large.

## Hard constraints
- Minimal viable change; don't touch unrelated files.
- Don't change public APIs / add deps without justification.
- No production resources.
- On a requirement contradiction → STOP and report.

## Why this shape (decision log)
> The D0–D6 selection procedure that derived this loop's shape.
- **D0** (loop) — runnable checks exist: loop-constructor's evals/run_all.mjs (53-case linter+renderer battery), the linter on a produced design, and grep for the D6 section — done is machine-decidable
- **D1** (staged (2 seams)) — seam1: the D6 guidance is authored and the battery stays green (handed to dogfood); seam2: a lint+render-valid completeness-first design exists (handed to the attacker review). Each upstream check runs before the downstream stage exists
- **D2** (plan_execute_verify on all three) — each stage is a deliberate change whose failure mode is a plausible-looking edit that doesn't actually hold (hollow guidance; a mislabeled design; a harden that breaks the battery) — verify-each-step against a thorough check is the pattern
- **D3** (on_the_loop) — reversible (git); the thorough battery + lint discriminate well; the one subjective bit (does the cadence guidance read true) is gated by the fresh-reader + the attacker review at the gate, not per-iteration approval
- **D4** (medium) — strictly sequential — guidance -&gt; dogfood -&gt; review each depend on the prior artifact; no independent fan-out (the 3 attacker lenses are internal to stage 3)
- **D5** (low caps (2/2/2, outer 2) + escalate) — completeness-first: a pass is meant to land; on exhaustion escalate to a human rather than thrash
- **D6** (completeness_first) — the iteration boundary is expensive (each pass reloads the whole skill's contract + reasons about cross-file consistency) and the check is thorough/slow (a 53-case battery + a full attacker review), so few long complete passes beat many shallow ones — tuned: low max_iterations, large per-stage scope, plan_execute_verify, thorough checks

## Stages (run in dependency order; each stage is its own gated loop)
> After a stage gate goes green, commit/checkpoint it; on a loopback, reset to the last green checkpoint rather than editing forward.

### Stage 1 — `author_d6_guidance`  ·  pattern: plan_execute_verify

- **Done when:** the D6 'Iteration profile (cadence)' section + its knob-mapping table exist in references/loop-selection.md, and D6 is wired into SKILL.md (SELECT), references/loop-design-shape.md (selection_log), and assets/fresh-reader-checklist.md (cadence box); loop-constructor's evals/run_all.mjs still passes 53/53
- **Check (run — the gate):** `node skills/loop-constructor/evals/run_all.mjs && grep -q "D6 — Iteration profile" skills/loop-constructor/references/loop-selection.md`  ·  expect: `pass`
- **Falsifiable when:** the D6 section is missing (grep fails) OR a doc edit breaks the linter/renderer battery (run_all goes red)
- **Passing-but-wrong (ruled out):** D6 exists as an empty heading with no knob-mapping content — grep passes but the guidance is hollow. Ruled out by the dogfood stage (which exercises the actual completeness-first knob choices) + the fresh-reader cadence box
- **Depends on:** — (entry stage)  ·  admit downstream only once this check passes.
- **Stop:** max 2 iterations; failure branches:
    - the D6 guidance cannot be authored without breaking the existing battery
- **On failure:** escalate.

### Stage 2 — `dogfood_verify`  ·  pattern: plan_execute_verify

- **Done when:** a completeness-first loop-design (this file) lints PASS and renders a runbook, and the fresh-reader cadence box holds — its knobs actually match the claimed completeness_first profile (low max_iterations, plan_execute_verify, thorough checks), not a mislabel
- **Check (run — the gate):** `node skills/loop-constructor/scripts/lint_loop_design.mjs .loop/loop-constructor-d6.design.json && node skills/loop-constructor/scripts/render_loop_doc.mjs .loop/loop-constructor-d6.design.json --out .loop --slug loop-constructor-d6 --force`  ·  expect: `pass`
- **Falsifiable when:** the design omits a required field or carries an always-green check (linter non-zero), OR the renderer refuses to emit a runbook (render non-zero) — the DoD covers BOTH lint AND render
- **Passing-but-wrong (ruled out):** the design lints green while being a MISLABELED completeness-first (high caps + retry + smoke checks) — the linter cannot catch this (D6 is guidance-only, not a linted field). Ruled out only by the fresh-reader cadence box + the attacker contract-gameability lens (this is the honest residual of the guidance-only choice)
- **Depends on:** `author_d6_guidance`  ·  admit downstream only once this check passes.
- **Stop:** max 2 iterations; failure branches:
    - the design cannot be made to both lint AND honestly exhibit the completeness-first knobs
- **On failure:** loop back to `author_d6_guidance`.

### Stage 3 — `attacker_review_harden`  ·  pattern: plan_execute_verify

- **Done when:** a 3-lens vince-attacker review (contract gameability of D6 + this concrete design + the whole loop-constructor structure) is run; every PROVEN structural defect is fixed and the evals/run_all.mjs battery stays 53/53; any unfixable finding (e.g. D6 not linter-enforced) is recorded as an honest residual
- **Check (run — the gate):** `node skills/loop-constructor/evals/run_all.mjs && test -f .loop/loop-constructor-d6/attacker-findings.md`  ·  expect: `pass`
- **Falsifiable when:** a hardening edit breaks the linter/renderer battery (run_all red), OR no attacker-findings.md artifact exists (the review wasn't recorded) — the check now covers both the regression AND that the review produced a recorded artifact
- **Passing-but-wrong (ruled out):** the battery is green but the attacker found a structural gap the 53 cases don't cover — exactly why the attacker (not just the battery) is this stage's gate; the DoD requires proven findings fixed-or-recorded, judged by the maker/checker, not by the battery alone
- **Depends on:** `dogfood_verify`  ·  admit downstream only once this check passes.
- **Stop:** max 2 iterations; failure branches:
    - an attacker finding requires a schema/linter change the user scoped OUT (guidance-only) — record as residual, do not silently expand scope
- **On failure:** loop back to `dogfood_verify`.

## Stop conditions (overall)
- **Success:** D6 is authored + wired into all four docs, the 53-case battery stays green and goldens still lint, this completeness-first design lints+renders with knobs that honestly match the profile, and the 3-lens attacker review's proven findings are fixed or recorded as residuals
- **Max iterations (budget cap):** 2
- **Failure branches:**
  - any stage exhausts its max_iterations without passing its check
  - the attacker proves D6 is purely cosmetic (changes no knob) and it cannot be made real within guidance-only scope
- **Escalate to a human when:**
  - a real fix requires a schema/linter change (scoped OUT: guidance-only) — surface to the user
  - the completeness-first guidance and the existing D0-D5 procedure turn out irreconcilable

## Terminal states
- **SUCCESS** — D6 is authored + wired into all four docs, the 53-case battery stays green and goldens still lint, this completeness-first design lints+renders with knobs that honestly match the profile, and the 3-lens attacker review's proven findings are fixed or recorded as residuals → done.
- **STOPPED_UNMET** — a max-iteration cap is hit or a failure branch fires: stop, report the last green stage and the unmet gate, hand back the partial diff, do NOT merge. Never relax a gate to force a pass.
- **ESCALATE** — one of the escalate triggers above fires: hand to a human.

## Human placement
- on_the_loop

## Checker (separate, fresh-context)
- a fresh-context reviewer sees only the produced docs + this design + the per-stage checks; flags: (a) a D6 section that reads concrete but doesn't actually change any knob choice, (b) this design being a mislabeled completeness-first, (c) loop-constructor structural gaps (linter blind spots, selection holes, persist-contract). Correctness/requirements only

## Risk guards
- **the cadence is cosmetic — D6 reads nice but changes no actual knob choice (reward hacking on prose)** → the dogfood stage forces D6 to drive real knob settings; the fresh-reader cadence box + attacker lens-1 attack a mislabel
- **mislabel trap — a design claims completeness-first while carrying high caps + retry + smoke checks (D6 not linter-enforced, by the guidance-only choice)** → the fresh-reader cadence box is the enforcement point; recorded as an explicit honest residual rather than pretended-closed
- **scope creep — an attacker finding tempts a schema/linter change the user scoped OUT** → guidance-only is a hard boundary; such findings are recorded as residuals + escalated, never silently built
- **regression to the existing battery from doc edits** → every stage re-runs the 53-case evals/run_all.mjs; goldens must still lint

## Harness primitives
- git per-file edits to skills/loop-constructor/ for reversible work
- loop-constructor's own evals/run_all.mjs (53-case linter+renderer battery) as the thorough per-stage check
- the linter + render_loop_doc on a produced design as the dogfood gate
- vince-attacker (3 lenses) as the independent structural review; its validators gate the records
- external scratch ledger .loop/loop-constructor-d6/run-state.md across context resets
